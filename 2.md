# SmartPM-MindSpore-AI-Demo 项目概述

## 1. 项目定位

这是一个面向产品需求分析场景的 AI 演示项目，目标是把一段自然语言需求文本转成更结构化、更可评审的分析结果。项目把传统规则校验、轻量分类模型、示例检索和大模型生成组合在一起，形成一条完整的“需求质检”流水线。

它更像一个可演示的原型系统，而不是一个已经产品化的生产平台。项目重点展示的是：

- 如何对原始需求做基础质量审计
- 如何用 MindSpore 风格的轻量模型做需求类别判断
- 如何用 RAG 思路补充相似案例
- 如何把结果整理成 Given / When / Then 验收表达
- 如何在前端把分析过程和结果可视化

## 2. 整体架构

项目采用前后端分离结构：

- 前端：`React + TypeScript + Vite + Tailwind CSS + Recharts`
- 后端：`FastAPI + Pydantic`
- AI 能力层：
  - 规则审计：基于本地 `quality_rules.json`
  - 分类能力：基于 `MindSporeService`
  - 检索能力：基于本地种子知识的简化 RAG
  - 文本生成：`MockService` 或 `DeepSeekService`

整体调用链路如下：

1. 用户在前端输入需求文本
2. 前端调用后端 `POST /api/analyze`
3. 后端并行/串行组织四类能力：
   - `AuditService`：识别模糊词和风险词
   - `MindSporeService`：预测需求类别
   - `RagService`：检索历史相似优质需求
   - `MockService` 或 `DeepSeekService`：生成 GWT 和改写建议
4. 后端把结果整合为统一响应
5. 前端将结果展示为评分、雷达图、风险列表、分类结果、RAG 示例、GWT 结果和改写建议

## 3. 后端职责

后端入口在 `backend/app/main.py`，主要暴露 4 个接口：

- `GET /api/examples`：返回示例需求，供前端随机填充
- `POST /api/analyze`：完整分析入口
- `POST /api/mindspore/predict`：单独做分类预测
- `POST /api/rag/search`：单独做示例检索

核心模块说明：

- `config.py`
  - 管理项目名、版本号、API 前缀
  - 控制是否使用 `USE_MOCK`
  - 管理 DeepSeek 相关配置

- `schemas.py`
  - 定义请求与响应结构
  - 统一约束 `audit`、`mindspore`、`rag`、`gwt` 四类结果

- `services/audit_service.py`
  - 从 `data/quality_rules.json` 读取规则
  - 对需求文本中的模糊词做命中检测
  - 生成扣分结果、风险项和雷达图数据

- `services/mindspore_service.py`
  - 优先加载 `mindspore_model/predict.py` 中的预测器
  - 若运行环境缺少 MindSpore、模型未训练或加载失败，则自动降级到内置启发式规则
  - 返回类别包括：`clear`、`ambiguous`、`incomplete`、`untestable`

- `services/rag_service.py`
  - 从 `data/seed_knowledge.json` 读取种子案例
  - 通过非常轻量的字符匹配方式计算相似度
  - 选出 Top-K 相似需求及其 GWT 示例
  - 这是一个简化版 RAG，更偏向演示思路，不是向量数据库方案

- `services/mock_service.py`
  - 在不开启真实大模型时返回固定格式的模拟结果
  - 便于本地演示和前端联调

- `services/deepseek_service.py`
  - 将 RAG 检索结果拼进 Prompt
  - 调用 DeepSeek 接口生成质量评分、风险信息、GWT 和改写版本
  - 出现网络、鉴权或响应格式异常时，会返回兜底结果，避免前端崩溃

## 4. MindSpore 模型部分

`backend/app/mindspore_model` 是项目里的轻量分类模块，职责是把需求文本分成 4 类：

- `clear`
- `ambiguous`
- `incomplete`
- `untestable`

主要实现方式：

- `feature_extractor.py`
  - 使用极简字符级特征提取
  - 将文本转成固定长度向量

- `model.py`
  - 定义一个非常轻量的全连接分类网络

- `dataset.py`
  - 负责读取训练数据并封装为数据集

- `train.py`
  - 使用 `training_samples.json` 训练模型
  - 输出模型文件和特征提取器

- `predict.py`
  - 加载训练好的 checkpoint 与 extractor
  - 对输入文本做推理并返回类别和置信度

从代码实现看，这部分更强调“可以跑通完整训练-预测链路”，而不是追求高精度。它适合作为 MindSpore 接入示例。

## 5. 数据资源

项目内置了三类核心数据：

- `backend/app/data/quality_rules.json`
  - 模糊词/风险词规则库
  - 用于规则审计与扣分说明

- `backend/app/data/seed_knowledge.json`
  - 历史优质需求和对应 GWT 示例
  - 用于简化版 RAG 检索

- `backend/app/data/training_samples.json`
  - 需求分类训练样本
  - 用于 MindSpore 分类模型训练

这些数据都是本地 JSON 文件，说明项目当前是“离线样例驱动”的设计，便于演示和学习，但数据规模有限。

## 6. 前端职责

前端位于 `frontend`，整体是一个单页分析看板。

核心文件说明：

- `src/App.tsx`
  - 负责页面整体布局、请求触发和结果状态管理
  - 组织输入区、流程区、架构区和结果区

- `src/api/client.ts`
  - 封装 Axios 请求
  - 定义前端使用的响应类型

- `src/components/RequirementInput.tsx`
  - 提供需求输入框、随机示例和分析按钮

- `src/components/WorkflowView.tsx`
  - 以步骤形式展示分析进度
  - 当前流程分为规则审计、MindSpore 分类、RAG 检索、大模型生成 4 步

- `src/components/QualityDashboard.tsx`
  - 展示总评分和雷达图

- `src/components/RiskList.tsx`
  - 展示命中的模糊风险词和原因

- `src/components/MindSporePanel.tsx`
  - 展示分类标签、置信度和是否为降级结果

- `src/components/RetrievedExamples.tsx`
  - 展示 RAG 检索出的相似案例

- `src/components/GWTResult.tsx`
  - 展示 Given / When / Then 结构化验收结果

- `src/components/ImprovementPanel.tsx`
  - 展示大模型改写后的更清晰需求文本

从交互上看，这个前端主要承担“解释 AI 结果”的职责，而不是复杂业务操作页面。

## 7. 典型分析流程

当用户输入一条原始需求后，系统会按下面顺序输出结果：

1. 找出需求里的模糊表达，比如“尽快”“稳定”“大概”之类
2. 给出一个初始质量评分和风险列表
3. 判断这条需求更像是清晰、模糊、不完整还是不可测试
4. 从本地知识样例中找出相似的优质需求和 GWT 写法
5. 结合检索上下文，由 Mock 或 DeepSeek 生成：
   - 更完整的质量评分
   - 风险项说明
   - Given / When / Then 验收表达
   - 改写后的更高质量需求文本

## 8. 项目特点

这个项目有几个比较鲜明的特点：

- 能力链路完整
  - 虽然每个模块都比较轻量，但从输入到结果输出是完整闭环

- 对演示场景友好
  - 即使没有 MindSpore 环境或真实大模型，也能通过 fallback 和 mock 跑通

- 便于教学和二次开发
  - 目录清晰，模块边界明确，适合继续替换更强的模型或检索方案

- 前端可视化较完整
  - 不只是返回 JSON，而是把分析过程和结果拆成多个可理解面板

## 9. 当前实现的边界

从当前代码来看，这个项目更适合作为 Demo 或课程样例，主要边界包括：

- RAG 不是基于向量库，而是简单字符匹配
- 分类模型较轻量，训练数据规模也有限
- 大模型结果依赖外部接口稳定性
- 规则库、样本库和种子知识目前都是本地静态 JSON
- 还没有看到数据库、鉴权、任务队列、日志审计等生产级设施

换句话说，这个项目的价值主要在于“把需求分析 AI 工作流跑通并展示出来”，而不是提供工业级精度或大规模服务能力。

## 10. 适合的后续扩展方向

如果继续演进，这个项目比较自然的升级方向包括：

- 将 RAG 替换为 Embedding + 向量数据库
- 扩充训练样本并升级分类模型
- 把规则库和知识库迁移到数据库或可配置后台
- 增加接口测试、前端状态处理和错误提示
- 支持更多需求质量维度，例如一致性、可追踪性、优先级合理性
- 对接真实产品流程，如 PRD 评审、测试用例生成、缺陷预防

## 11. 总结

`SmartPM-MindSpore-AI-Demo` 是一个围绕“需求质量分析”构建的前后端一体化 AI Demo。它把规则引擎、轻量分类模型、简化 RAG 和大模型生成组合起来，输出评分、风险、相似案例、GWT 验收表达和优化后的需求文本。

如果把它看成一个学习样板，它的结构已经很完整；如果把它看成一个产品原型，它也已经具备清晰的扩展路径。
